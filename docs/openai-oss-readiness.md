# OpenAI OSS Program Readiness

OSS Intake Doctor should help the open-source community first.

OpenAI OSS program readiness is a later outcome, not the first reason to build.

The project should apply only after it has public proof. A new local repo with no maintainer usage is not strong enough yet.

## Why This Project Fits The Category

OSS Intake Doctor is maintainer automation.

It helps with:

- Issue triage.
- Issue-template quality.
- Missing-information reduction.
- Duplicate surfacing.
- Support routing.
- Security-sensitive report routing.
- Maintainer workflow quality.

The fit is strongest when framed as a trust-first maintainer workflow tool, not as another AI issue triage bot.

## Competitive-Aware Framing

Do not claim that no one else has issue triage.

GitHub already documents AI-powered issue intake, and several Marketplace actions classify issues, suggest labels, detect duplicates, and post comments.

The stronger claim is:

```text
OSS Intake Doctor gives maintainers a no-cost, no-external-API intake quality toolkit: local issue-form audits before submission, then dry-run issue analysis before public bot comments.
```

This matters because cautious OSS maintainers may not want to send public issue content to external AI APIs, manage paid keys, or allow bots to write comments before seeing evidence.

## What We Need Before Applying

Before applying to any OSS support program, the project should have:

- Public repository.
- Clear README.
- License.
- Security policy.
- Contribution docs.
- Passing tests.
- Safe dry-run workflow.
- Real maintainer feedback.
- Evidence from at least one pilot.
- Clear competitive positioning.
- Public proof that the no-cost approach is useful.

## Evidence To Collect

Useful evidence:

- Number of issues analyzed.
- Missing-info rate.
- Template audit score.
- Issue form improvements made before pilot.
- Duplicate suggestions and maintainer-confirmed usefulness.
- Support questions routed.
- Security-sensitive reports routed safely.
- Maintainer quotes or feedback.
- Before/after examples.
- False-positive examples and how they were handled.
- Setup time for a new maintainer.
- Proof that no external APIs or paid services were required.

## Strong Application Framing

```text
OSS Intake Doctor is a no-cost, no-external-API open-source maintainer automation tool that reduces duplicate, incomplete, support-style, and security-sensitive GitHub issue burden before maintainers waste time.
```
