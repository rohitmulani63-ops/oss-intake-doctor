# Positioning

## Plain-English Position

OSS Intake Doctor is a zero-cost, no-external-API maintainer intake quality toolkit for GitHub issues.

It audits issue forms before submission, then checks whether a new issue has enough information, looks like support, touches security-sensitive topics, or may duplicate existing issues before maintainers spend time on it.

## What It Is

- A local, transparent issue-quality checker.
- A local issue-template audit.
- A dry-run first GitHub Action.
- A maintainer signal tool.
- A safer step before teams adopt AI-heavy automation.
- A future home for contributor-reviewed Repro Packs.

## Project Boundaries

- Local rules first.
- Free core workflow.
- Dry-run before public comments or labels.
- Maintainer control over issue actions.
- No hosted service required.
- No paid API required.
- Maintainers remain the final reviewers.

## Competitive Truth

The GitHub issue triage category already exists.

GitHub documents AI-powered issue intake, and there are Marketplace tools that classify issues, detect duplicates, post comments, and apply labels with LLM APIs.

OSS Intake Doctor can credibly claim:

```text
Most triage tools are AI/API/action-after-submission. OSS Intake Doctor is the no-cost, privacy-first, explainable intake quality layer maintainers can trust before they let automation write to their repos.
```

## Best One-Line Pitch

```text
OSS Intake Doctor helps open-source maintainers reduce incomplete, duplicate, support-style, and security-sensitive GitHub issues with zero-cost issue-template audits and no-external-API dry-run checks.
```

## Best Short Pitch

```text
Open-source maintainers waste time asking for missing versions, reproduction steps, logs, and duplicate context. AI-heavy triage tools exist, but many require API keys, paid usage, external issue sharing, or more automation than cautious maintainers want.

OSS Intake Doctor starts smaller and safer: issue-template audits, transparent local rules, conservative duplicate hints, support/security routing, config validation, and dry-run summaries. It gives maintainers signal before it asks for trust.
```

## Strategic Wedge

1. No API key required.
2. No paid model required.
3. No external issue-content sharing.
4. Dry-run first.
5. Transparent local rules.
6. Local issue-form audit before pilot installs.
7. Conservative duplicate suggestions with reasons.
8. Config validation before write actions.
9. Security-sensitive routing without public detail expansion.
10. Repro Pack direction for better bug reports.
11. Maintainer trust before automation.

## OpenAI OSS Program Angle

The strongest application story is not that OSS Intake Doctor uses AI.

The stronger story is that it helps open-source maintainers create cleaner issue queues, safer intake workflows, and measurable triage evidence. If pilots show maintainers save time, the project becomes a credible candidate for broader OSS support and future optional AI assistance.
